## Issue 1 - Brain win-rate falls back to default

Root cause:
- `DailyBrain` asks `_get_recent_win_rate()` for the last configured `recent_win_rate_lookback_trades`, currently 15 in `config/settings.yaml:104`; the hard minimum is 5 closed P&L rows before a source is considered usable (`algotrader/strategy_selector/brain.py:988`, `algotrader/strategy_selector/brain.py:1000`, `algotrader/strategy_selector/brain.py:1046`).
- On 2026-05-04, IBKR mode wired a `BrokerLedger` into Brain (`algotrader/orchestrator.py:100`, `algotrader/orchestrator.py:216`). The ledger query groups closed round trips by `perm_id, symbol` and requires at least one non-null `realized_pnl` (`algotrader/tracking/broker_ledger.py:392`, `algotrader/tracking/broker_ledger.py:418`, `algotrader/tracking/broker_ledger.py:420`).
- `broker_fills` in `data/journal/trades.db` had 0 rows and 0 closed groups. Before the patch, Brain returned `None` immediately when ledger history was under 5 rows, so it never reached the populated strategy-journal fallback.

Journal evidence:
- `trades` has 63 closed rows with non-null `realized_pnl`; no closed rows are missing P&L.
- Last 15 closed journal rows span `2026-04-10T14:08:49Z` to `2026-04-30T13:31:13Z`, with 13 wins and 2 losses, so the expected win rate is `0.867`.
- The strategy journal is being persisted by `StrategyBase.record_trade()` with `exit_time` and `realized_pnl` (`algotrader/strategies/base.py:158`, `algotrader/strategies/base.py:194`, `algotrader/strategies/base.py:195`, `algotrader/strategies/base.py:202`), and `TradeJournal.record_trade()` inserts those fields into SQLite (`algotrader/tracking/journal.py:75`, `algotrader/tracking/journal.py:83`, `algotrader/tracking/journal.py:98`, `algotrader/tracking/journal.py:99`).

Implemented fix:
- Changed `_get_recent_win_rate()` so insufficient broker-ledger history logs `recent_win_rate_ledger_insufficient` and falls through to the strategy journal instead of forcing the caller to use `recent_win_rate_fallback` (`algotrader/strategy_selector/brain.py:1000`, `algotrader/strategy_selector/brain.py:1020`, `algotrader/strategy_selector/brain.py:1030`).
- Verified against the real DB after the patch: with an empty broker ledger, Brain now returns `0.8666666666666667` from the strategy journal.

Test coverage:
- Added `test_brain_uses_journal_when_ledger_insufficient_but_journal_has_history` in `tests/unit/test_broker_ledger.py`.
- Updated the insufficient-ledger test to use an explicit empty journal so it still verifies the true fallback-to-default case.

Safe to merge:
- Yes. This is scoped to Brain source selection. It does not alter journal or broker-ledger persistence and only uses the existing fallback source when the preferred source is too sparse.

## BrokerLedger follow-up - why `broker_fills` is empty

Findings:
- The wiring is present in IBKR mode. `Orchestrator` creates `BrokerLedger(db_path="data/journal/trades.db", ib=self._ibkr_conn.ib)` and passes it into Brain (`algotrader/orchestrator.py:100`, `algotrader/orchestrator.py:216`).
- `BrokerLedger` subscribes to the live IB events `execDetailsEvent` and `commissionReportEvent` (`algotrader/tracking/broker_ledger.py:106`, `algotrader/tracking/broker_ledger.py:107`, `algotrader/tracking/broker_ledger.py:108`). The installed `ib_async 2.1.0` package confirms those callback signatures are `execDetailsEvent(trade, fill)` and `commissionReportEvent(trade, fill, report)`.
- `_on_exec_details()` inserts each live fill into `broker_fills`, and `_on_commission_report()` updates commission and realized P&L (`algotrader/tracking/broker_ledger.py:115`, `algotrader/tracking/broker_ledger.py:164`, `algotrader/tracking/broker_ledger.py:204`, `algotrader/tracking/broker_ledger.py:228`).
- Option orders are submitted through `IBKRExecutor.submit_mleg_order()`, which calls `ib.placeOrder()`, registers the returned trade, and logs `ibkr_mleg_order_submitted` (`algotrader/execution/ibkr_executor.py:829`, `algotrader/execution/ibkr_executor.py:911`, `algotrader/execution/ibkr_executor.py:912`, `algotrader/execution/ibkr_executor.py:916`).

Evidence:
- On 2026-05-04 the ledger did initialize and wire events (`data/logs/algotrader.log.2026-05-04:19`, `data/logs/algotrader.log.2026-05-04:20`).
- The same log has zero `ibkr_mleg_order_submitted`, `options_mleg_submitted`, `premium_entry`, `trade_recorded`, `broker_fill_recorded`, or `broker_fill_record_failed` events.
- Direct SQLite checks show `broker_fills` has 0 rows, `trades` has 0 May 4 entries, `trades` has 0 May 4 exits, and the latest strategy-journal exit is `2026-04-30T13:31:13.118173+00:00`.
- The pre-rotation log contains earlier option submissions and strategy-journal records, including April 29-30 activity, but no `broker_ledger_initialized` event. That history predates the live BrokerLedger wiring in the available logs.

Conclusion:
- I did not find evidence that live IBKR fills were silently dropped on 2026-05-04. The empty table is explained by no submitted/fillable orders after BrokerLedger was wired, plus no historical execution backfill.
- The missing capability is startup reconciliation/backfill from IBKR executions, not a low-risk one-line fix. The repo has no `reqExecutions` usage, and `BrokerLedger` currently records only future live callback events plus explicit test/manual `record_fill()` calls (`algotrader/tracking/broker_ledger.py:252`).

Proposed future fix:
- Add a deliberate BrokerLedger backfill path that calls IBKR execution history on startup for a bounded window, normalizes those fills through the same insert/update methods, and logs a summary. This should be reviewed separately because execution-history filtering, duplicate handling, combo-leg shape, and commission timing affect ledger truth.

## Issue 2 - Transient `drawdown_pct: 100.0` at startup

Root cause:
- Brain does not compute drawdown from an equity high-water mark. It reads `drawdown_pct` directly from `data/state/broker_snapshot.json` (`algotrader/strategy_selector/brain.py:1076`, `algotrader/strategy_selector/brain.py:1103`).
- The actual portfolio snapshot formula is guarded against zero peak equity: `drawdown_pct = ((peak - equity) / peak * 100) if peak > 0 else 0` (`algotrader/tracking/portfolio.py:28`, `algotrader/tracking/portfolio.py:70`, `algotrader/tracking/portfolio.py:96`).
- The startup ordering allowed Brain to read a stale on-disk broker snapshot before the current process wrote a fresh one. The premarket Brain decision runs in `_refresh_premarket_intelligence()` (`algotrader/orchestrator.py:749`) and the fresh dashboard snapshot is written afterward (`algotrader/orchestrator.py:788`). The normal trading-loop reset/write happens later (`algotrader/orchestrator.py:460`, `algotrader/orchestrator.py:463`).
- The log matches that ordering: `adaptive_risk_computed` reported `drawdown_pct: 100.0` at `data/logs/algotrader.log.2026-05-04:81`; `portfolio_day_reset` with live equity `104198.0` happened immediately after at line 84; the next Brain event at line 131 read `drawdown_pct: 0.0`.

Implemented fix:
- `DailyBrain` now records its initialization time (`algotrader/strategy_selector/brain.py:175`) and ignores broker snapshots whose `timestamp` predates the current Brain process (`algotrader/strategy_selector/brain.py:1096`).
- It also ignores missing/invalid snapshot timestamps and non-finite or negative drawdown values (`algotrader/strategy_selector/brain.py:1081`, `algotrader/strategy_selector/brain.py:1085`, `algotrader/strategy_selector/brain.py:1105`).

Test coverage:
- Added `test_current_drawdown_ignores_stale_startup_snapshot` in `tests/unit/test_brain.py`.
- Added `test_current_drawdown_reads_fresh_broker_snapshot` so valid fresh snapshots still drive adaptive risk.

Safe to merge:
- Yes. This is a defensive Brain read guard. It does not modify risk management, portfolio tracking, or journal logic.

## Issue 3 - Concurrent-session Error 10197 and modeled-credit fallback

Original behavior:
- The 2026-05-04 log contains 90 IBKR `Error 10197` events: 40 SPY and 50 QQQ, from `2026-05-04T10:19:05Z` through `2026-05-04T13:29:05Z`.
- In the provided 2026-05-04 file, I found no `options_live_credit_unavailable`, `premium_entry`, or order-construction events. The unsafe behavior was confirmed in code and appears in earlier logs.
- `OptionsPremiumStrategy` has a `require_live_credit_estimate` config flag and the checked config already sets it to `true` (`algotrader/strategies/options_premium.py:122`, `config/strategies/options_premium.yaml:34`).
- Before the patch, the flag was not enforced. When `_estimate_live_credit_per_contract()` returned `None`, the strategy only changed the log level, left `selected_credit_per_contract` set to the modeled estimate, and continued to submit the spread.

Implemented hardening:
- `OptionsPremiumStrategy` now honors the existing flag. If `live_credit_per_contract is None` and `_require_live_credit_estimate` is true, it logs `options_live_credit_unavailable` at warning level with `skip_reason="live_credit_required"` and continues past the entry (`algotrader/strategies/options_premium.py:474`, `algotrader/strategies/options_premium.py:476`, `algotrader/strategies/options_premium.py:481`).
- If the flag is false, the modeled-credit fallback remains available for deliberate degraded-mode operation and logs `reason="falling_back_to_modeled_credit_estimate"` (`algotrader/strategies/options_premium.py:486`, `algotrader/strategies/options_premium.py:490`).

Config action for Marco:
- No config change is needed for the hardened behavior because `require_live_credit_estimate: true` is already set.
- Operationally, the IBKR conflict still needs to be resolved outside the bot by ensuring no competing live/paper market-data session is logged into the same paper account during the trading window.

Test coverage:
- Added `test_options_premium_requires_live_credit_when_configured`, which verifies a missing live credit skips the entry, emits `skip_reason="live_credit_required"`, and submits no multi-leg order (`tests/unit/test_strategy_candidates.py:410`).
- Added `test_options_premium_can_fall_back_to_modeled_credit_when_live_credit_not_required`, which verifies the explicit degraded-mode branch still submits using modeled credit when the flag is false (`tests/unit/test_strategy_candidates.py:468`).
- Updated the QQQ strike-rounding test to provide a live-credit estimate so that test continues to cover rounding rather than quote-availability policy (`tests/unit/test_swing_trading.py:212`, `tests/unit/test_swing_trading.py:250`).

Safe to merge:
- Yes. The default config already requires live credit; this patch makes the existing policy effective while preserving the modeled-credit path behind an explicit false flag.
